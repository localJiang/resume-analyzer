# AI 编码规则

> **适用于：** 所有在此项目上工作的 AI Agent
> **结合阅读：** `CLAUDE.md`、`AGENTS.md`、`memory/*.md`

## 核心指令

### 1. 先理解再行动
- 在编写代码前阅读相关的 `memory/` 文件
- 在创建新模式前阅读现有代码模式
- 若需求不明确，提出澄清性问题
- 绝不假设 — 对照文档进行验证

### 2. 遵循约定
- `CLAUDE.md` 中的所有规则均为强制性的
- 匹配现有代码风格（命名、结构、模式）
- 不引入未文档化的新模式
- 若约定未被文档记录，则匹配现有代码库的风格

### 3. 编写完整的代码
- 所有代码必须达到生产就绪标准 — 无占位符、无未关联 Issue 的 TODO
- 每个函数必须有错误处理
- 每个组件必须处理加载中、空数据、错误状态
- 每个类型必须是显式的 — 无 `any`

### 4. 测试一切
- 为新代码编写测试（单元测试 + 集成测试视情况而定）
- 运行现有测试以确保无回归
- Bug 修复必须包含回归测试
- 测试错误情况，而不仅是正常路径

### 5. 记录变更
- 当约定或架构变更时更新 `memory/` 文件
- 当任务状态变更时更新 `docs/TASKS.md`
- 为重要的架构决策添加 ADR
- 当端点变更时更新 API 文档

### 6. 正确提交
- 约定式提交格式：`type(scope): description`
- 小而聚焦的提交（每次提交一个逻辑变更）
- 绝不提交密钥、调试代码或注释掉的代码
- 绝不提交生成的文件（`node_modules`、`.next`、`dist`）

### 7. 保持在范围内
- 仅实现任务/需求中的内容
- 未经许可不重构无关代码
- 未经询问不添加“锦上添花”的功能
- 如果发现范围外的问题，创建 Issue — 不要现在就修复它

## 禁止操作

```
❌ 修改 .git/ 目录或 Git 历史
❌ 未经确认运行破坏性命令（rm -rf、DROP TABLE、git push --force）
❌ 提交密钥（API 密钥、令牌、密码）
❌ 在 TypeScript 中使用 `any` 类型
❌ 跳过新功能或 Bug 修复的测试
❌ 留下无跟踪 Issue 的 TODO 注释：// TODO(#123)
❌ 修改生成的代码（Prisma client、GraphQL 类型）
❌ 更换包管理器（坚持项目已选择的那一个）
❌ 使用内联样式 — 使用 Tailwind 工具类
❌ 直接提交到 `main` — 始终使用功能分支
❌ 在提交中留下 console.log 或调试代码
```

## 必需操作

```
✅ 编写自文档化的代码，使用清晰的变量/函数名
✅ 保持函数小巧（< 40 行）
✅ 保持文件聚焦（< 300 行，然后拆分）
✅ 添加错误边界和有意义的错误信息
✅ 使用 TypeScript 严格模式特性
✅ 提交前运行 Lint 和格式化
✅ 在客户端和服务器端都验证用户输入
✅ 处理所有状态：加载中、空数据、错误、成功、禁用
✅ 使 UI 组件可访问（键盘、屏幕阅读器、对比度）
```

## Agent 工作流

每个 AI Agent 必须遵循以下循环：

```
1. 理解
   阅读 memory/、docs/、CLAUDE.md → 理解上下文

2. 计划
   将任务分解为步骤 → 识别受影响的文件 → 为复杂工作编写计划

3. 实现
   编写代码 → 添加类型 → 处理错误 → 编写测试

4. 验证
   运行测试 → 运行 Lint → 自审查检查清单

5. 文档
   更新 memory/ → 更新 docs/ → 更新 tasks/ → 使用正确信息提交
```

## 质量门禁（声明完成前）

```
□ 类型正确（无 `any`、无不安全的类型转换）
□ 所有失败模式均有错误处理
□ 所有边界处均有输入校验
□ 测试通过（新增 + 现有）
□ Lint 通过
□ 格式化通过
□ 无调试代码或 console.log
□ 无硬编码的密钥
□ 加载/空数据/错误状态已处理（UI）
□ 可访问（UI）
□ 文档已更新
□ 提交信息正确
```
