# 产品需求文档：Resume Analyzer

> **版本：** 1.1
> **日期：** 2026-07-01
> **状态：** 草案

---

## 执行摘要

Resume Analyzer 是一款 **AI 驱动的简历分析工具**。用户上传简历文件（PDF/DOCX/TXT），系统通过大语言模型对其进行多维度分析——内容完整度、关键词匹配、格式规范、语言表达——并给出逐条改进建议，帮助求职者提升简历竞争力。

## 问题陈述

### 当前现状
求职者制作简历时普遍面临三个痛点：
1. **不知道简历好不好** — 投递后石沉大海，缺乏客观反馈。大多数人凭感觉写简历，不清楚 HR 和 ATS（申请人追踪系统）到底看重什么
2. **专业简历顾问太贵** — 一次简历修改服务动辄 300–1000 元，对初入职场或频繁求职的人来说成本过高
3. **网上建议太泛** — 搜到的简历技巧千篇一律（"要量化成果""保持一页"），缺少针对具体简历的具体建议

### 目标用户
主要面向 **主动求职者**，包括应届毕业生和有 1–5 年经验的职场人。

| 用户画像 | 描述 | 核心需求 |
|---------|-------------|--------------|
| **应届毕业生** | 缺乏工作经验，简历内容单薄，不知道如何包装实习和项目经历 | 补充缺失的关键模块，优化有限内容的表达方式 |
| **初级职场人（1–5年）** | 有工作内容但不会量化成果，简历冗长或结构混乱 | 提炼亮点、量化结果、优化关键词匹配 ATS |
| **海投求职者** | 用同一份简历投多个岗位，回复率低 | 针对目标岗位的关键词和技能要求检查简历匹配度 |

### 价值主张
**为什么是现在？** 三个趋势交汇：
1. **LLM 能力成熟** — GPT-4/Claude 级别的模型已经能给出高质量的文本分析和修改建议，成本下降到可以商业化的水平
2. **求职市场竞争加剧** — 一份简历平均被浏览 6 秒，优化简历从"加分项"变成"必选项"
3. **用户已习惯 AI 工具** — ChatGPT 培养了用户"让 AI 帮我改文档"的心智，无需市场教育

相比通用 AI 对话，Resume Analyzer 的差异在于：**针对简历场景预置了分析框架（维度、评分标准、行业最佳实践），用户不用自己写 prompt，上传即得结构化分析报告**。

## 目标与成功指标

### 业务目标
1. **提升简历质量** — 帮助求职者通过 AI 分析发现简历中的问题并改进，使简历通过初筛的概率提升 30% 以上
2. **降低求职成本** — 让用户无需依赖付费简历顾问，即可获得专业级的简历诊断和建议

### 成功指标（KPI）

| 指标 | 目标值 | 衡量方式 |
|--------|--------|-------------------|
| 用户采用率 | 上线 3 个月内注册用户 ≥ 1000 | 后台数据统计 |
| 用户留存率 | 次月留存率 > 30% | 数据分析 |
| 核心转化 | 用户完成首次简历分析的比例 > 60% | 埋点跟踪 |
| 性能 | 简历分析接口 p95 < 3s | APM 监控 |
| 满意度 | 分析结果有用性评分 ≥ 4.0/5.0 | 分析完成后内嵌评分 |

### 非目标（不在范围内）
1. **不做简历模板/在线编辑器** — 不提供可视化拖拽排版、字体配色等简历美化功能
2. **不做职位匹配推荐** — 不抓取招聘网站数据，不做简历与职位的自动匹配和投递
3. **不做 HR 端/企业端** — MVP 阶段只面向求职者，不涉及招聘方的人才筛选功能

## 功能特性

### 功能优先级
- **P0（MVP）：** 上线必备
- **P1（V1.1）：** 高价值，上线后尽快交付
- **P2（V2）：** 重要但不紧急
- **P3（未来）：** 锦上添花

### P0 — 最小可行产品

#### 功能 1：简历上传与解析
- **用户故事：** 作为求职者，我希望上传我的简历文件（PDF/DOCX/TXT），系统自动解析文本内容，以便开始分析
- **验收标准：**
  - 假设用户在 `/dashboard` 页面，当他拖拽 PDF 文件到上传区时，文件被接受并开始上传
  - 假设用户上传了一个 .docx 文件，当文件大小超过 5MB 时，系统提示"文件过大，请上传 5MB 以内的文件"
  - 假设用户上传了 PDF 文件，当解析完成后，系统自动跳转至分析页面 `/analysis/[id]`
  - 假设用户上传了 .jpg 文件，系统提示"不支持的格式，请上传 PDF/DOCX/TXT"

#### 功能 2：AI 简历分析
- **用户故事：** 作为求职者，我希望 AI 对我的简历进行多维度分析，以便知道具体哪里需要改进
- **验收标准：**
  - 假设简历解析完成，当 AI 开始分析时，用户看到骨架屏加载态（含维度名称轮播提示）
  - 假设分析完成，当用户查看报告时，可以看到：总评分（0-100 数字 + 环形进度）、四个维度的分类 Tab、每个维度下的逐条问题和建议
  - 假设用户对某条建议不满意，可以点击"重新分析"按钮触发重新分析

#### 功能 3：分析历史管理
- **用户故事：** 作为求职者，我希望能查看我过去上传和分析过的简历记录，以便对比改进效果
- **验收标准：**
  - 假设用户有分析历史，当进入 `/dashboard` 时，下方展示历史卡片列表（简历名 + 评分 + 分析时间）
  - 假设用户点击历史卡片，跳转至该次分析报告 `/analysis/[id]`
  - 假设用户没有任何分析记录，看到空状态引导（插图 + "还没有分析记录" + 上传按钮）

## 用户流程

### 主流程：上传简历 → 查看分析

```
步骤 1：用户进入首页 / →
步骤 2：点击「开始分析」CTA → 跳转注册页 /auth/signup →
步骤 3：注册完成后自动登录 → 进入工作台 /dashboard →
步骤 4：拖拽或点击上传简历文件 →
步骤 5：系统解析文件 → 显示骨架屏加载态 →
步骤 6：AI 分析完成 → 展示分析报告 /analysis/[id] →
步骤 7：用户查看评分和逐条改进建议
```

### 页面路由

| 路由 | 页面 | 核心组件 |
|------|------|---------|
| `/` | 首页（落地页） | Hero + 功能卡片 + CTA |
| `/auth/signin` | 登录 | Input, Label, Button, Card |
| `/auth/signup` | 注册 | Input, Label, Button, Card |
| `/dashboard` | 工作台 | 上传区 + 分析历史卡片列表 |
| `/analysis/[id]` | 分析报告 | 评分区 + Tabs + 问题卡片 |

### 边缘情况
- 当用户未登录直接访问 `/dashboard` 时 → 重定向到 `/auth/signin`
- 当分析失败（AI 返回错误）时 → 显示错误提示 +「重新分析」按钮，不丢已上传的文件
- 当用户上传不支持的文件格式时 → Toast 提示"不支持此格式，请上传 PDF/DOCX/TXT"
- 当文件超过大小限制时 → 在上传前就拦截并提示，不浪费上传时间
- 当用户访问不存在的分析 ID 时 → 显示 404 页面，带返回工作台的链接
- 当网络断开时 → 显示离线提示，网络恢复后自动重试

## 设计要求

### 设计基准

参照 BOSS 直聘的设计风格：**干净 / 专业 / 克制 / 呼吸感 / 即时反馈**。通过 shadcn/ui 组件体系落地，用 CSS 变量统一控制全局主题。

### 主题配置（`globals.css`）

```css
@layer base {
  :root {
    /* 背景 — 暖灰底色，卡片纯白 */
    --background: 240 10% 97%;       /* #F7F8FA */
    --foreground: 240 6% 10%;        /* #1A1A1A 正文 */
    --card: 0 0% 100%;               /* #FFFFFF */
    --card-foreground: 240 6% 10%;

    /* 品牌色 — 蓝，传达 AI/专业/可信 */
    --primary: 221 83% 53%;          /* #2563EB */
    --primary-foreground: 0 0% 100%;

    /* 辅助色 */
    --secondary: 240 10% 96%;
    --secondary-foreground: 240 6% 10%;
    --muted: 240 10% 96%;
    --muted-foreground: 240 4% 46%;  /* #8C8C8C 辅助文字 */
    --accent: 142 71% 45%;
    --accent-foreground: 0 0% 100%;

    /* 描边主导，阴影为辅（BOSS直聘卡片风格） */
    --border: 220 13% 91%;           /* #EBEDF0 */
    --ring: 221 83% 53%;

    /* 圆角系统 */
    --radius: 0.5rem;                /* 8px  — 按钮/输入框 */
    --radius-lg: 0.75rem;            /* 12px — 卡片 */
  }
}
```

### 页面 × 组件映射

| 页面 | shadcn 组件 | 说明 |
|------|-----------|------|

### 核心组件设计要点

**卡片（Card）**
- 描边主导：`border border-border` + 极浅阴影 `shadow-[0_1px_2px_rgba(0,0,0,0.04)]`
- hover 时边框变色而非放大：`hover:border-primary/30 transition-colors duration-200`
- 空状态/上传区用虚线描边：`border-dashed border-2`

**按钮（Button）**
- 主操作：`variant="default"`（品牌色实心）
- 次要操作：`variant="outline"`（描边）
- Ghost：`variant="ghost"`（纯文字，用于导航/菜单）
- 所有交互态覆盖：default / hover / active / focus / disabled / loading

**上传区**
- 虚线边框默认态（`border-dashed border-border`）
- 拖入时边框变实线 + 品牌色 + 背景微亮（`border-solid border-primary bg-primary/5`）
- 过渡动画 `transition-all duration-200`

**Tabs（分析报告分类切换）**
- `TabsList` 背景 `bg-muted`，选中态品牌色下划线
- `TabsTrigger` 圆角 `rounded-md`，字体 15px / 500
- 切换时底部指示条滑动动画（CSS `translateX transition`）

**Skeleton（骨架屏）**
- 分析加载时替代 spinner：灰色占位块 + 扫光动画
- 布局与真实页面一致：评分数字 → Tab 栏 → 问题卡片垂直排列

**空状态**
- `Card className="border-dashed py-16 text-center"` + 简洁插图 + 引导文案 + CTA 按钮
- 不用 emoji 代替图标

### 交互细节

| 场景 | 行为 |
|------|------|
| 评分出现 | 数字从 0 count-up 至目标值（~800ms） |
| Tab 切换 | 内容区 fade 过渡，非闪变 |
| 骨架屏 | 分析中展示维度名称轮播文案 |
| 拖拽反馈 | 边框变色 + 背景微亮（200ms ease） |
| 通知 | sonner Toast：上传成功/分析完成/错误提示 |

### UI 红线

- ❌ 不用纯黑文字（`#000`），最深用 `#1A1A1A`
- ❌ 同一页面不超过 3 种字号（标题/正文/辅助）
- ❌ loading 状态不能是白屏 — 必须有骨架屏或进度指示
- ❌ 空状态不能空白 — 必须有引导文案 + CTA
- ❌ 不用 emoji 代替图标，统一用 Lucide SVG 图标
- ❌ 按钮不只有颜色区分 — 次要按钮用描边+文字，确保色觉无障碍
- ✅ 所有交互状态覆盖：default / hover / active / focus / disabled / loading

### 无障碍
- WCAG 2.1 AA 最低标准（shadcn/ui 自带键盘导航和屏幕阅读器支持）
- 所有 shadcn 组件已内置 focus ring、role 属性、键盘交互

## 风险与缓解措施

| 风险 | 可能性 | 影响程度 | 缓解措施 |
|------|-----------|--------|------------|
| LLM API 调用延迟过高 | 中 | 高 | 设置 30s 超时，超时后提示用户重试；骨架屏 + 进度文案降低焦虑感 |
| LLM 分析质量不稳定 | 中 | 高 | 预置结构化 prompt + few-shot 示例约束输出格式；加用户评分反馈闭环 |
| 文件解析不准确（PDF 排版复杂） | 高 | 中 | 先用成熟的解析库（pdf-parse/mammoth），解析结果展示给用户确认 |
| 用户上传敏感信息 | 低 | 高 | 文件处理完即删，不持久存储原始简历内容；隐私政策明确告知 |
