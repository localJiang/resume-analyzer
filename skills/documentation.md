# 技能：文档

## 描述
创建和维护面向人类和 AI Agent 的项目文档。

## 适用场景
- 编写 README 和入门文档
- 记录 API 端点
- 编写架构决策记录（ADR）
- 创建代码注释和 JSDoc
- 维护面向 AI 上下文的 memory/ 文件
- 编写发布说明和变更日志

## 输入
- 待文档化的源代码
- 待记录的架构决策
- 待描述的 API 端点
- 待编写的用户指南

## 输出
- 更新后的 `docs/` 文件
- 更新后的 `memory/` 文件
- 公共 API 的 JSDoc 注释
- API 文档（OpenAPI）
- 发布说明

## 约束
- 同时面向人类和 AI Agent 编写
- 将文档放在所描述代码的附近
- 代码变更时更新文档（文档过时比没有文档更糟）
- 使用一致的 Markdown 格式
- 文档中的代码示例必须是可运行/有效的
- 不记录显而易见的事情（`// 将 i 递增 1`）

## 文档类型

| 类型 | 位置 | 受众 | 更新触发条件 |
|------|----------|----------|---------------|
| 项目概览 | `README.md`、`memory/project.md` | 所有人 | 新功能、技术变更 |
| 架构 | `docs/ARCHITECTURE.md`、`memory/architecture.md` | 开发者、AI | 架构变更 |
| API 文档 | OpenAPI/Swagger、`memory/api.md` | 前端、外部 | 新增/修改端点 |
| 代码文档 | JSDoc、行内注释 | 开发者 | 代码变更 |
| 决策 | `memory/architecture.md`（ADR） | 开发者、AI | 做出关键决策 |
| 任务 | `docs/TASKS.md` | PM、开发者 | 状态变更 |
| 路线图 | `docs/ROADMAP.md` | 所有人 | 里程碑变更 |

## 最佳实践
1. **文档即代码** — 所有文档纳入版本控制
2. **保持就近** — `memory/` 文件是 AI 上下文，保持其准确性
3. **代码示例需可验证** — 测试你的代码片段
4. **解释为什么而非是什么** — 代码已经展示了是什么；文档应解释为什么
5. **渐进式细节** — 概览 → 详情 → 参考
6. **随变更更新** — 将“更新文档”纳入你的完成定义
7. **可搜索** — 在所有文档中使用一致的术语
