# 技能：调试

## 描述
系统性地识别、诊断和修复应用中的缺陷。

## 适用场景
- 调查 Bug 报告
- 修复失败的测试
- 排查性能问题
- 调试生产事件
- 追踪竞态条件
- 调试异步/时序问题

## 输入
- 带有复现步骤的 Bug 报告
- 错误日志或堆栈跟踪
- 测试失败输出
- 性能指标

## 输出
- 根因分析
- 附带回归测试的 Bug 修复
- 预防性建议
- 更新后的文档

## 约束
- 修复前必须先复现 — 不做猜测性修复
- 每个 Bug 必须编写回归测试
- 修复根因而非症状
- 在代码库中搜索相同的 Bug 类型
- 不引入新 Bug — 修复后运行完整测试套件

## 调试流程

### 1. 复现
- 按照 Bug 报告的确切步骤操作
- 使用不同数据、浏览器、环境尝试
- 判断是一致性还是间歇性问题

### 2. 隔离
- 检查受影响区域最近的 Git 变更
- 二分搜索：注释掉一半代码 → 问题是否仍存在？
- 缩小到最小复现案例

### 3. 诊断
- 前端：控制台、网络面板、React DevTools、断点
- 后端：服务器日志、curl/Postman、数据库查询
- 数据库：EXPLAIN ANALYZE、检查迁移、检查索引

### 4. 修复
- 编写能复现 Bug 的失败测试 → 验证测试失败
- 实现修复 → 验证测试通过
- 运行完整测试套件 → 验证无回归
- 思考：代码库中其他地方是否存在相同类型的 Bug？

### 5. 预防
- 什么流程/工具可以在早期发现此问题？
- 添加校验、类型约束或测试
- 若假设有误，更新文档

## 最佳实践
1. **仔细阅读错误信息** — 答案往往就在堆栈跟踪中
2. **优先检查最简单的解释** — 通常不是编译器 Bug
3. **一次只改一个东西** — 否则你不知道是什么修复了它
4. **适时休息** — 新鲜的眼睛比疲劳的眼睛更快找到 Bug
5. **橡皮鸭调试法** — 大声解释问题
6. **搜索类似问题** — 检查 Git 历史、Issue 跟踪器、Stack Overflow
