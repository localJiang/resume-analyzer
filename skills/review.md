# 技能：代码审查

## 描述
审查代码变更的正确性、风格、安全性和性能。

## 适用场景
- 审查 Pull Request
- 合并前的质量门禁
- 变更的安全审计
- 架构合规性检查
- 通过代码审查进行指导

## 输入
- Pull Request 差异
- 相关 Issue 或任务
- CLAUDE.md 编码标准
- 架构文档

## 输出
- 审查结论：✅ 已批准 / 💬 评论 / 🔄 需要修改
- 逐行反馈
- 整体评估
- 安全问题（如有）

## 约束
- 审查必须在 24 小时内完成
- 严重问题必须在合并前修复
- 绝不批准包含 `any` 类型的代码
- 绝不批准没有错误处理的代码
- 始终检查安全问题
- 保持建设性：提出修复建议，而非仅指出问题
- 正确性优先，风格次之

## 审查检查清单

```
🔴 严重（必须修复）
□ 安全漏洞（XSS、注入、密钥暴露）
□ 缺少对预期失败的错误处理
□ 业务逻辑错误
□ 缺少认证/授权检查
□ N+1 数据库查询

🟡 重要（应当修复）
□ 缺少 TypeScript 类型（any 使用）
□ 缺少输入校验
□ 缺少新行为的测试
□ 命名违规（CLAUDE.md 约定）
□ 复杂代码无注释/解释

🟢 优化建议（锦上添花）
□ 小的风格改进
□ 性能微优化
□ 替代方案建议
□ 文档改进
```

## 最佳实践
1. **审查逻辑而非仅审查语法** — 它真的解决了问题吗？
2. **具体明确** — 引用确切的文件和行号
3. **解释原因** — 不只是“修复这个”，而是“修复这个因为 X 可能发生”
4. **表扬好的方案** — 正向反馈很重要
5. **不要吹毛求疵** — 自动化工具处理格式问题；专注于实质
6. **限时审查** — 不要在一个 PR 上花费数小时
