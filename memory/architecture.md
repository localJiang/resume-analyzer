## 架构风格

**Feature First（功能优先）**

按功能模块组织代码——每个功能文件夹包含自己的组件、hooks、服务、类型和测试。相关代码集中管理，便于添加、修改或删除功能。

### 目录结构

```
src/
├── app/                    # 应用路由和入口点
├── features/               # 功能模块（功能优先）或领域模块（DDD）
│   ├── auth/
│   │   ├── components/
│   │   ├── hooks/
│   │   ├── services/
│   │   ├── types.ts
│   │   └── index.ts
│   ├── users/
│   └── ...
├── components/             # 共享 UI 组件
│   ├── ui/                 # 基础 UI 组件
│   ├── layout/             # 布局组件
│   └── common/             # 共享功能组件
├── lib/                    # 共享工具函数
│   ├── api.ts              # API 客户端
│   ├── auth.ts             # 认证辅助
│   ├── date.ts             # 日期工具

└── styles/                 # 全局样式
```

## 数据流

```
用户操作 → 组件 → Hook/服务 → API 客户端 → 后端 → 数据库
                                          ↓
                                    React Query 缓存
                                          ↓
                                    组件重新渲染
```

### 状态管理策略

| 状态类型 | 解决方案 | 位置 |
|-----------|----------|----------|
| URL 参数 | Next.js Router | URL |
| 服务端状态 | React Query / SWR | 缓存 |
| 表单状态 | React Hook Form | 组件 |
| UI 状态 | useState | 组件 |
| 全局状态 | Zustand / Context | `src/stores/` |

## API 设计

### RESTful 约定

```
GET    /api/users          → 列出用户（分页）
GET    /api/users/:id      → 根据 ID 获取用户
POST   /api/users          → 创建用户
PUT    /api/users/:id      → 更新用户（完整）
PATCH  /api/users/:id      → 更新用户（部分）
DELETE /api/users/:id      → 删除用户（软删除）
```

### 响应格式

```typescript
// 成功
{
  "data": T | T[],
  "meta"?: {
    "page": number,
    "pageSize": number,
    "total": number,
  }
}

// 错误
{
  "error": {
    "code": string,
    "message": string,
    "details"?: Array<{ field: string; message: string }>
  }
}
```

## 认证流程

使用 **Supabase Auth**：

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│  客户端   │────▶│  认证服务器    │────▶│  数据库   │
│           │     │  (Supabase Auth)    │     │           │
│  登录     │     │  验证         │     │  Users    │
│  会话     │     │  签发 Token   │     │  表       │
│  JWT/PKCE │     │  刷新         │     │           │
└──────────┘     └──────────────┘     └──────────┘
```

### Token 策略
- 访问令牌：短期有效（15 分钟），存储在内存中
- 刷新令牌：长期有效（7 天），存储在 HTTP-only Cookie 中
- 刷新时：轮换刷新令牌

## ADR 记录

### ADR-001：选择 Next.js
**日期：** 2026-06-30
**状态：** 已接受

**背景：** 需要为本项目选择前端框架。
**决策：** 使用 Next.js，理由：
1. 成熟的生态系统和社区
2. 出色的 TypeScript 支持
3. 强大的性能特性
4. 团队熟悉

**后果：**
- 可访问丰富的组件库生态系统
- 需要谨慎管理 SSR/CSR 边界
- 打包体积优化很重要

### ADR-002：选择 Next.js API 路由
**日期：** 2026-06-30
**状态：** 已接受

**背景：** 需要为 API 开发选择后端框架。
**决策：** 使用 Next.js API 路由，理由：
1. 强大的类型和验证支持
2. 异步优先设计
3. 自动生成 API 文档
4. 良好的性能特性

### ADR-003：选择 PostgreSQL 配合 Prisma
**日期：** 2026-06-30
**状态：** 已接受

**背景：** 需要为数据持久化选择数据库和 ORM。
**决策：** 使用 PostgreSQL 配合 Prisma，理由：
1. 类型安全的数据库访问
2. 声明式结构定义
3. 自动迁移生成
4. 良好的查询性能
