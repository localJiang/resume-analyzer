# /review — 代码审查工作流

## 用途
审查代码的正确性、风格、安全性和性能。

## 使用场景
- 合并 Pull Request 之前
- `/implement` 完成之后
- 被要求审查他人的代码时
- 生产环境部署之前

## 工作流

### 第 1 步：理解上下文
1. 阅读 PR 描述和关联的 Issue
2. 理解变更应该做什么
3. 阅读该 PR 实施的计划/任务
4. 检查所有验收标准是否均已覆盖

### 第 2 步：审查检查清单

#### 正确性
- [ ] 代码是否做到了它声称要做的事？
- [ ] 边缘情况是否已处理？（null、undefined、空、错误）
- [ ] 所有分支的逻辑是否正确？
- [ ] 是否存在竞态条件或时序问题？

#### 代码质量
- [ ] 是否遵循 `CLAUDE.md` 的命名约定？
- [ ] 函数是否小巧且单一职责？
- [ ] 是否有应提取的重复代码？
- [ ] TypeScript 类型是否正确（无 `any`）？
- [ ] 导入是否组织正确？

#### 安全性
- [ ] 是否有硬编码的密钥或 API 密钥？
- [ ] 用户输入是否在客户端和服务器端都已校验？
- [ ] SQL 查询是否参数化（无注入）？
- [ ] 受保护路由是否检查了认证？
- [ ] 是否存在 XSS 漏洞？

#### 性能
- [ ] 是否存在 N+1 数据库查询？
- [ ] 是否避免了不必要的重渲染？
- [ ] 大列表是否分页？
- [ ] 昂贵计算是否已记忆化？

#### 测试
- [ ] 测试是否覆盖了正常路径？
- [ ] 测试是否覆盖了错误情况？
- [ ] 测试是否覆盖了边缘情况？
- [ ] 测试描述是否清晰且具有描述性？

#### UI/UX（如适用）
- [ ] 加载、空、错误状态是否已处理？
- [ ] 组件是否可访问（键盘、屏幕阅读器）？
- [ ] 响应式设计是否正确？
- [ ] 错误信息是否用户友好？

### 第 3 步：提供反馈
- 要具体：引用文件和行号
- 要建设性：建议如何修复，而不仅仅指出哪里有问题
- 要友善：聚焦代码，而非作者
- 区分：🔴 必须修复 vs 🟡 应该修复 vs 🟢 锦上添花

### 第 4 步：决定
- ✅ **批准** — 可以合并
- 💬 **评论** — 有小建议，不阻塞合并
- 🔄 **请求修改** — 必须修复后才能合并

## 审查输出格式

```markdown
## 审查摘要
**结论：** 🔄 请求修改
**已审查文件数：** 5
**发现问题：** 2 个严重、3 个轻微

## 严重问题 🔴

### 1. `user.service.ts:42` 缺少错误处理
**问题：** 数据库查询可能抛出异常，但没有 try/catch
**修复：** 用 try/catch 包裹，返回类型化的 Result 类型
**建议：**
```typescript
try {
  const user = await db.user.findUnique(...);
} catch (error) {
  throw new DatabaseError('获取用户失败', error);
}
```

## 轻微问题 🟡

### 1. `auth.ts:15` 中的魔法字符串
使用常量代替内联字符串

## 锦上添花 🟢

### 1. 考虑提取重复的校验逻辑
该模式出现了 3 次，可以提取为共享工具
```

## 约束条件
- 审查必须在 24 小时内完成
- 严重问题必须修复后才能合并
- 绝不要批准包含 `any` 类型的代码
- 绝不要批准没有错误处理的代码
- 批准前始终检查安全问题
