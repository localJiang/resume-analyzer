# /plan — 规划工作流

## 用途
分析需求并创建可执行的实施计划。

## 使用场景
- 开始新功能之前
- 需求需要澄清时
- 进行架构变更之前
- 分解复杂任务时

## 工作流

### 第 1 步：收集上下文
1. 阅读 `memory/project.md` — 理解项目标识
2. 阅读 `memory/architecture.md` — 理解约束条件
3. 阅读 `specs/` 中的相关功能规范
4. 阅读受影响区域的现有代码

### 第 2 步：分析需求
1. 明确需要构建什么
2. 与 PM Agent 澄清不明确的地方
3. 定义验收标准
4. 识别边缘情况和错误条件

### 第 3 步：设计方案
1. 识别受影响的文件和组件
2. 设计数据流和状态管理
3. 设计 API 契约（如需新接口）
4. 考虑性能、安全、无障碍

### 第 4 步：分解任务
1. 分解为小的、独立的任务
2. 按依赖关系排序任务
3. 估算复杂度（S/M/L）
4. 识别风险和缓解措施

### 第 5 步：编写计划
1. 在 `tasks/` 目录下创建计划文档
2. 更新 `docs/TASKS.md` 添加新任务
3. 在 `memory/` 文件中记录决策
4. 与相关 Agent 分享以获取反馈

## 输出格式

```markdown
# 计划：[功能名称]

## 概述
[2-3 句话的概述]

## 需求
- [需求 1]
- [需求 2]

## 技术设计
### 受影响的文件
- `src/features/...` — [变更内容]

### 数据流
[描述或图示]

### API 变更
- [新增/修改的接口]

### 状态变更
- [新增/修改的状态]

## 任务
| # | 任务 | 复杂度 | 依赖项 |
|---|------|-----------|--------------|
| 1 | ... | S | — |
| 2 | ... | M | 1 |

## 风险
- [风险] → 缓解措施

## 测试计划
- [需要测试的内容]
```

## 约束条件
- 计划必须可在 1 周工作量内完成
- 如果更大，请拆分为多个计划
- 每个计划必须包含测试策略
- 计划必须引用现有的架构决策
