# /implement — 实施工作流

## 用途
按照项目标准执行实施任务。

## 使用场景
- `/plan` 获批之后
- 实施 `docs/TASKS.md` 中的任务时
- 修复 Bug 时
- 重构代码时

## 工作流

### 第 1 步：准备
1. 阅读计划或任务描述
2. 阅读相关的 `memory/` 文件
3. 检查 `CLAUDE.md` 了解编码规范
4. 创建功能分支：`feature/<任务描述>`

### 第 2 步：实施（红-绿-重构）
1. **先写测试**（如果可以独立测试）
2. **编写最小化代码** 使测试通过
3. **重构** — 清理代码，添加类型，提高可读性
4. **运行测试** — `npm run test`
5. **运行 Lint** — `npm run lint`
6. **运行格式化** — `npm run format`

### 第 3 步：自查
在标记完成之前，请验证：
- [ ] 类型正确（无 `any`，无不经验证的 `as` 类型断言）
- [ ] 错误处理覆盖所有失败模式
- [ ] 边界处的输入校验
- [ ] UI 组件的加载/空/错误状态
- [ ] 无障碍属性已添加
- [ ] 无 console.log 或调试代码
- [ ] 无硬编码的密钥或 URL
- [ ] 导入顺序正确
- [ ] 已删除死代码

### 第 4 步：提交
```bash
git add <files>
git commit -m "<type>(<scope>): <描述>"
```

### 第 5 步：更新文档
- 如果约定或架构有变化，更新 `memory/` 文件
- 更新 `docs/TASKS.md` 中的任务状态
- 如果 API 有变化，更新 `docs/SPEC.md`

## 代码质量检查清单

```typescript
// 每次提交前，验证：
✅ 函数小巧（< 40 行）
✅ 文件聚焦（< 300 行）
✅ 类型显式（无 `any`）
✅ 错误已处理（无未捕获的 Promise）
✅ 导入已整理（6 个分组，见 CLAUDE.md）
✅ 命名遵循规范（见 CLAUDE.md）
✅ 测试已通过（npm run test）
✅ Lint 已通过（npm run lint）
✅ 无注释掉的代码残留
✅ 无没有 Issue 编号的 TODO：TODO(#123)
```

## 约束条件
- 绝不要直接提交到 `main` 或 `develop`
- 绝不要跳过新功能的测试
- 绝不要使用 `any` 类型
- 绝不要在提交中留下调试代码
- 推送前始终运行完整测试套件
