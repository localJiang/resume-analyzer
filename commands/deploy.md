# /deploy — 部署工作流

## 用途
安全地将应用部署到目标环境。

## 使用场景
- PR 获批并合并之后
- 发布新版本时
- 回滚有问题的部署时

## 环境

| 环境 | 分支 | URL | 用途 |
|-------------|--------|-----|---------|
| **预览** | `feature/*` | `<pr-number>.preview.{{DOMAIN}}` | PR 审查 |
| **预发布** | `develop` | `staging.{{DOMAIN}}` | 上线前测试 |
| **生产** | `main` | `{{DOMAIN}}` | 线上用户 |

## 工作流

### 第 1 步：部署前检查清单

- [ ] 所有测试通过（`npm run test`）
- [ ] Lint 通过（`npm run lint`）
- [ ] 类型检查通过（`npm run typecheck`）
- [ ] 构建成功（`npm run build`）
- [ ] 代码审查已批准
- [ ] PR 已合并到目标分支
- [ ] 数据库迁移已就绪（`npx prisma migrate status`）
- [ ] 环境变量已配置
- [ ] 构建产物中无密钥

### 第 2 步：部署

```bash
# 预览环境（PR 自动触发）
# 通过 GitHub Actions 自动部署

# 预发布环境（合并到 develop 自动触发）
# 通过 GitHub Actions 自动部署

# 生产环境（手动触发或合并到 main）
# 通过 Vercel 控制台或 CLI：
npx vercel deploy
npx vercel deploy --prod
```

### 第 3 步：部署后验证

- [ ] 健康检查接口返回 200：`GET /api/health`
- [ ] 数据库迁移已成功执行
- [ ] 核心用户流程正常工作（冒烟测试）
- [ ] 错误率没有飙升
- [ ] 响应时间在目标范围内（p95 < 200ms）
- [ ] 静态资源加载正常
- [ ] SSL/TLS 证书有效

### 第 4 步：监控（前 30 分钟）

- 监控错误追踪（Sentry）
- 监控性能指标
- 监控服务器日志
- 检查用户反馈
- 做好回滚准备

## 回滚流程

```bash
# 如果部署导致问题：

# 1. 通过 Vercel 控制台：
#    - 选择上一个成功的部署
#    - 点击"回滚"

# 2. 通过 CLI：
vercel rollback

# 3. 验证回滚：
#    - 健康检查
#    - 核心功能
#    - 错误率恢复正常

# 4. 通知：
#    - 在 Slack/Discord 中通知团队
#    - 创建事故 Issue
#    - 24 小时内进行复盘
```

## 发布检查清单

### 每次生产发布前
- [ ] 所有 P0 任务已完成并测试
- [ ] 安全审查已通过
- [ ] 已在负载下测试性能
- [ ] 数据库备份已确认
- [ ] 回滚计划已文档化
- [ ] 发布说明已编写
- [ ] 面向客户的变更已通知

## 环境变量检查清单

```bash
# 确认以下环境变量在目标环境中已设置：
DATABASE_URL          ✅ / ❌
AUTH_SECRET           ✅ / ❌
AUTH_URL              ✅ / ❌
NEXT_PUBLIC_API_URL   ✅ / ❌
ALLOWED_ORIGINS       ✅ / ❌
```

## 约束条件
- 绝不要在周五部署（除非是紧急热修复）
- 绝不要在没有回滚计划的情况下部署
- 始终先在预发布环境验证数据库迁移
- 生产部署后始终监控 30 分钟
- 始终向团队通报部署状态
