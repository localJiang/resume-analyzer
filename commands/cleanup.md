# /cleanup — 代码清理工作流

## 用途
系统性地清理和改善代码质量，不改变原有行为。

## 使用场景
- Sprint 清理日
- 重大重构之前
- 技术债务被识别时
- 功能完成后（进入下一个功能之前）
- 准备生产环境上线时

## 工作流

### 第 1 步：审计
运行以下检查并识别问题：

```bash
# Lint 检查
npm run lint

# 类型检查
npm run typecheck

# 查找未使用的依赖
npx depcheck

# 查找死代码
npx ts-prune

# 查找重复代码
npx jscpd src/

# 检查打包体积
npm run build -- --analyze
```

### 第 2 步：清理检查清单

#### 代码清理
- [ ] 删除注释掉的代码块
- [ ] 删除未使用的导入
- [ ] 删除未使用的变量和函数
- [ ] 删除调试用的 `console.log` 语句
- [ ] 删除没有 Issue 编号的 TODO 注释（或创建 Issue）
- [ ] 将重复代码整合为共享工具
- [ ] 删除无用的 CSS（使用 PurgeCSS 或类似工具）

#### 类型清理
- [ ] 将 `any` 替换为正确的类型
- [ ] 将类型断言（`as`）替换为正确的类型守卫
- [ ] 删除未使用的类型定义
- [ ] 整合重复的类型定义
- [ ] 为公共函数添加缺失的返回类型

#### 重构机会
- [ ] 将重复的模式提取为可复用的函数/hooks
- [ ] 拆分大文件（> 300 行）
- [ ] 拆分大函数（> 40 行）
- [ ] 简化复杂条件判断（提取为命名函数）
- [ ] 用命名常量替换魔法数字/字符串
- [ ] 改善变量/函数命名的清晰度

#### 依赖清理
- [ ] 删除未使用的 npm 包
- [ ] 更新过时的包（语义版本兼容范围内）
- [ ] 检查包重复（`npm dedupe`）
- [ ] 审计安全漏洞（`npm audit`）

#### 文件清理
- [ ] 删除空文件
- [ ] 删除未使用的资源（图片、字体等）
- [ ] 整理放置在错误目录的文件
- [ ] 删除 `.bak`、`.tmp`、`.old` 文件
- [ ] 如果 `.gitignore` 缺少模式则更新

### 第 3 步：应用修复

```bash
# 自动修复 Lint 问题
npm run lint:fix

# 自动格式化
npm run format

# 自动删除未使用的导出（先检查！）
npx ts-prune --skip '\.test\.' '\.spec\.'
```

### 第 4 步：验证
- [ ] 所有测试仍通过：`npm run test`
- [ ] 构建仍成功：`npm run build`
- [ ] 无新增 Lint 警告
- [ ] 手动冒烟测试核心功能

### 第 5 步：提交

```bash
git add .
git commit -m "chore: 清理代码

- 删除未使用的导入和死代码
- 修复 Lint 警告
- 重构 [模块] 中的重复代码
- 更新过时的类型定义"
```

## 清理前后指标

在清理前后跟踪以下指标：

| 指标 | 清理前 | 清理后 | 改善 |
|--------|--------|-------|-------------|
| Lint 警告数 | X | Y | -Z |
| `any` 类型数量 | X | Y | -Z |
| > 300 行文件数 | X | Y | -Z |
| 打包体积（KB） | X | Y | -Z |
| 测试覆盖率 % | X | Y | +Z |

## 约束条件
- 清理不得改变行为 — 只能改善结构和可读性
- 清理前后均需运行完整测试套件
- 以小而专注的提交进行清理 — 不要一个巨大的"清理"提交
- 不要在同一提交中同时进行重构和 Bug 修复
- 重大重构变更需经过审查
- 绝不要清理你不理解的代码
